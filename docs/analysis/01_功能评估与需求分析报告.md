## 二、功能缺口分析（GAP Analysis）

### 2.1 业务流程缺口

#### 缺口 1：采购管理模块仅有数据模型，无可用 UI
**现状**：`PurchasePlan`、`PurchaseRequest`、`PurchaseOrder`、`PurchaseOrderItem`、`PurchaseQuote` 模型已完整定义，但对应的 `/purchase` 路由文件（`purchases.py`）内容为空（188字节）。  
**影响**：采购流程完全无法操作，导致备件的采购申请→审批→下单→收货闭环断裂。  
**严重度**：🔴 高

#### 缺口 2：设备管理模块无操作界面
**现状**：`Equipment` 模型已定义，`equipment.py` 路由文件仅 188 字节（空壳）。  
**影响**：设备与备件的消耗关联无法维护，维修工单无法关联设备。  
**严重度**：🟡 中

#### 缺口 3：维修工单管理无操作界面  
**现状**：`MaintenanceOrder`、`MaintenanceTask`、`MaintenanceRecord`、`MaintenanceCost` 模型已定义，`maintenance.py` 路由仅 194 字节（空壳）。  
**影响**：备件与设备维修的消耗记录无法管理。  
**严重度**：🟡 中

#### 缺口 4：供应商管理缺少评价与合规管理
**现状**：`Supplier` 和 `SupplierEvaluation` 模型已定义，但无完整的 CRUD 路由。  
**影响**：供应商资质管理、绩效评价无法操作。  
**严重度**：🟡 中

#### 缺口 5：调拨单管理功能不完整
**现状**：`Transaction` 模型支持 transfer 类型，`transactions.py` 有调拨路由，但出入库模型（`InboundOrder`/`OutboundOrder`）的 `transfer_order_id` 字段无关联单据表。  
**影响**：跨仓库调拨无独立单据追踪，无法查看调拨历史。  
**严重度**：🟡 中

#### 缺口 6：报表统计模块为空壳
**现状**：`reports.py` 仅 182 字节，无实际内容。前端无报表页面。  
**影响**：管理层无法获取经营分析数据（库存价值、消耗趋势、采购成本等）。  
**严重度**：🔴 高

#### 缺口 7：移动端/扫码功能入口缺失
**现状**：后端有条形码搜索 API，但无专门的扫码入库/出库移动端入口。  
**影响**：现场操作人员需在 PC 端操作，效率低。  
**严重度**：🟠 中高

#### 缺口 8：调拨申请审批流缺失
**现状**：出库单有审批字段（`approval_required`/`approval_status`），但无独立的审批流引擎，审批结果无法通知申请人。  
**影响**：多仓库调拨、大额出库无审批控制。  
**严重度**：🟡 中

### 2.2 数据完整性缺口

| 问题 | 现状 | 影响 |
|------|------|------|
| `MaintenanceRecord.spare_parts_used` 为 Text 类型 | 无法关联 SparePart 外键，无法自动扣减库存 | 维修消耗无法追溯 |
| `InboundOrder`/`OutboundOrder` 无多明细设计 | 单单只能关联一个备件（一对一） | 采购入库往往包含多种备件 |
| `Transaction` 与 `InboundOrder`/`OutboundOrder` 重叠 | 两套流水记录体系 | 数据冗余，统计混乱 |
| 批次（Batch）与库存记录（InventoryRecord）关联缺失 | `inventory_record` 无 `batch_id` 外键 | 先进先出无法实现 |

### 2.3 用户体验缺口

| 问题 | 严重度 |
|------|--------|
| React 前端仅覆盖仓库管理和备件管理，其余模块仍为 Jinja2 模板，UI 风格不统一 | 🟡 |
| 仪表盘（Dashboard）数据源指向旧的统计字段，与 InventoryRecord 联动数据不一致 | 🟡 |
| 移动端无响应式优化（WarehouseList.jsx 使用 Ant Design，SparePartList.jsx 使用 Bootstrap，混用 UI 库） | 🟠 |
| 批量操作缺少进度提示 | ⚪ |
| 搜索无防抖处理（每次键盘事件触发请求） | ⚪ |

### 4.3 系统架构影响分析

**对后端架构的影响：**
- 新增 P0 功能不改变现有数据库表结构（仅 F003 需新增 `transfer_order` 表）
- 新增 API 蓝图在 `register_blueprints()` 中注册，无架构耦合
- 所有新功能均沿用 Flask Blueprint + SQLAlchemy 架构模式

**对前端架构的影响：**
- 新页面统一加入 React Router，在 `App.jsx` 中追加路由
- 新服务文件在 `frontend/src/services/` 下按模块分文件
- 建议统一 UI 库（当前混用 Ant Design 和 Bootstrap），新增页面优先使用 React Bootstrap 保持一致

**对数据库的影响：**
- F001~F006 均可基于现有表结构实现
- F011（FIFO/FEFO）需要在 `inventory_record` 表增加 `batch_id` 外键（需 ALTER TABLE）
- 所有变更均向下兼容，不破坏现有数据

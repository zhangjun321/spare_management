## 一、模块现状梳理

### 1.1 备件管理模块（Spare Parts）

#### 菜单结构

```
备件管理
├── 备件列表（/spare_parts/）
│   ├── 搜索/筛选（part_code, name, category, supplier, status）
│   ├── 分页展示（默认每页20条）
│   └── 批量操作（导出CSV/Excel）
├── 备件详情（/spare_parts/<id>）
│   ├── 基础信息（编码/名称/规格/品牌/单价/单位）
│   ├── 库存信息（当前库存/最低/最高/安全/再订货点）
│   ├── 多图片管理（主图/缩略图/侧面图/详细图/电路图/透视图，共6种）
│   ├── 条形码生成（/spare_parts/<id>/generate_barcode）
│   └── AI填充信息（/spare_parts/<id>/ai_fill）
├── 新增/编辑备件（/spare_parts/new、/spare_parts/<id>/edit）
├── 备件分类管理（/spare_parts/categories）
└── 备件导出（/spare_parts/export/csv|excel）
```

#### 核心功能与业务流程

**备件生命周期**：
```
创建备件 → 设置库存阈值 → 关联供应商/分类 → 入库上架 
    → 日常出入库操作 → 实时库存跟踪 → 预警触发 → 补货/报废
```

**库存预警机制**：
- `SparePart.update_stock_status()` 判断状态（normal/low/out/overstocked）
- 状态变更时调用 `_trigger_alert()`，同时写 `Alert` + `Notification` 并发邮件
- 通知对象：系统管理员、仓库管理员（warehouse_manager角色）、采购员（purchaser角色）

**AI 辅助功能**：
- 条形码自动生成（基于 part_code 生成 EAN/Code128）
- AI 信息填充（通过外部 AI 服务补全备件技术参数、规格）
- 库存预测（尚未集成，仅有接口预留）

**数据模型（SparePart 关键字段）**：

| 字段分组 | 字段 | 说明 |
|---------|------|------|
| 标识 | part_code, barcode, name | 编码唯一，条形码唯一可空 |
| 库存 | current_stock, stock_status, min_stock, max_stock, safety_stock, reorder_point | 六维库存控制 |
| 图片 | image_url, thumbnail_url, side_image_url, detail_image_url, circuit_image_url, perspective_image_url | 6种视图 |
| 采购 | unit_price, last_purchase_price, last_purchase_date, currency | 价格追踪 |
| 技术 | technical_params (JSON), datasheet_url, warranty_period | 技术文档 |

**现存缺陷**：
- `SparePart.current_stock` 同时由 `InventoryRecord` 事件监听器（`_sync_spare_part_stock`）和 `TransactionService._apply_stock()` 双重维护，存在写入竞争风险
- `created_at/updated_at` 使用 `datetime.now`（非 UTC），与其他模型的 `datetime.utcnow` 不一致
- 无批次（Batch）维度的备件库存查询接口
- `safety_stock` 字段存在但 `update_stock_status()` 未使用，始终忽略安全库存触发逻辑

### 1.2 仓库管理模块（Warehouse Management）

#### 菜单结构

```
仓库管理
├── 仓库列表（/warehouses/）
│   ├── 仓库搜索/筛选
│   └── 仓库状态统计卡片
├── 仓库仪表盘（/warehouses/dashboard）
│   ├── 库存统计（总库存/备件种类/利用率）
│   ├── 入出库趋势图（Chart.js）
│   └── 库存预警列表
├── 仓库详情（/warehouses/<id>）
│   ├── 基础信息（名称/编码/类型/面积/容量/管理员）
│   ├── 7种图片管理（主图/缩略图/内部图/外观图/位置地图/布局图/缩略地图）
│   └── 预警配置（容量预警阈值/利用率阈值/自动预警开关）
├── 库区管理（WarehouseZone）
│   └── 库区分类（general/cold/hazardous/A/fast-moving）
├── 库位管理（WarehouseLocation）
│   ├── 货位状态（available/occupied/reserved/disabled）
│   └── AI 推荐入库库位
├── 入库管理（/warehouses/<id>/inbound）
│   ├── 创建入库单（InboundOrder）
│   ├── AI 推荐库位（/warehouses/inbound/<id>/ai_recommend）
│   ├── 执行入库（/warehouses/inbound/<id>/execute）
│   └── 撤销入库（/warehouses/inbound/<id>/rollback）
├── 出库管理（/warehouses/<id>/outbound）
│   ├── 创建出库单（OutboundOrder）
│   ├── 分配出库库位（FIFO/LIFO/FEFO策略）
│   ├── 执行出库
│   └── 撤销出库
├── 库存概览（/warehouses/<id>/inventory）
├── AI 分析（/warehouses/<id>/ai_analysis）
├── 质检标准管理
└── 质检管理
```

#### 核心功能与业务流程

**入库完整流程**：
```
创建入库单 → [可选] AI推荐库位 → 分配库位 
    → 执行入库（更新InventoryRecord + 同步SparePart.current_stock）
    → 更新仓库统计（触发after_insert事件）→ 更新InboundOrder状态
```

**出库完整流程**：
```
创建出库单 → 出库前库存校验 → [可选] 审批 
    → 自动分配库位（FIFO/LIFO/FEFO）→ 创建OutboundSourceLocation 
    → 执行出库（扣减InventoryRecord）→ 同步SparePart.current_stock
```

**三套并行仓库架构问题**：
- 旧版：`warehouses_bp`（`/warehouses`）→ 渲染 `warehouses/` 模板
- 中间版：部分页面渲染 `warehouse_new/` 模板（dashboard 路由使用）
- 新版（v3）：`warehouse_v3` 模板，已在 `__init__.py` 注册但具体蓝图未完整确认

这三套并行导致同一功能有多个入口，前端资源重复加载，维护成本高。

**数据同步机制**：
通过 SQLAlchemy ORM 事件（`after_insert/after_update/after_delete`）在 `InventoryRecord` 变化时，用原生 SQL 自动同步：
- `spare_part.current_stock`（SUM 所有仓库的 quantity）
- `warehouse.total_inventory`、`total_spare_parts`、`utilization_rate`

**现存缺陷**：
- `WarehouseService.get_warehouse_detail_statistics()` 引用 `InventoryRecord.total_value`（不存在，实际字段名为 `total_amount`），导致 500 错误
- `WarehouseService.get_total_inventory()` / `check_warehouse_inventory_alerts()` 使用旧版 `Transaction.warehouse_id`、`transaction_type` 字段（新版 Transaction 已重构为 `source/target_warehouse_id` + `tx_type`），导致查询异常
- `OutboundService._allocate_fifo/lifo/fefo()` 过滤条件 `status='normal'` 使用了不存在的字段（`InventoryRecord` 无 `status` 字段，应为 `stock_status`）
- `InboundService._analyze_part_and_recommend()` 末尾代码：`for location in rack.locations.filter_by(is_occupied=False).first()` 对 `first()` 返回的单个对象做迭代，逻辑错误

### 1.3 交易管理模块（Transaction Management）

#### 菜单结构

```
交易管理
├── 交易列表（/transactions/）
│   ├── 全类型汇总列表
│   ├── 多条件筛选（tx_type/status/日期范围/关键词）
│   └── 导出（CSV/Excel）
├── 入库单（/transactions/inbound）
│   └── 复用仓库入库模板
├── 出库单（/transactions/outbound）
│   └── 复用仓库出库模板
├── 调拨单（/transactions/transfer）
│   ├── 跨仓库调拨
│   └── 调用 TransactionService.transfer_spare_part()
├── 盘点差异（/transactions/inventory）
│   ├── 发起盘点（创建 InventorySnapshot）
│   └── 记录盘点差异（生成 inventory_adjust 类型 Transaction）
└── 交易详情（/transactions/<id>）
    ├── 审批操作（提交/审批/拒绝）
    └── 库存流水查看（InventoryLedger）
```

#### 核心功能与业务流程

**新版交易流程（TransactionService）**：
```
创建草稿(draft) → 提交(submitted) → [审批通过(approved) | 拒绝(rejected)]
   ↓ approved时触发
写入InventoryLedger（quantity_delta）→ 调用_apply_stock更新SparePart.current_stock
```

**四种交易类型**：

| tx_type | 中文 | source_wh | target_wh | 库存变化 |
|---------|------|-----------|-----------|---------|
| inbound | 入库 | 可空 | 目标仓库 | +delta |
| outbound | 出库 | 来源仓库 | 可空 | -delta |
| transfer | 调拨 | 来源仓库 | 目标仓库 | 来源-，目标+ |
| inventory_adjust | 盘点调整 | 任意仓库 | 任意仓库 | 可正可负 |

**双轨并行问题**：
- 旧版：`InboundOrder` + `OutboundOrder` + `InboundOrderItem/OutboundOrderItem` + `OutboundSourceLocation` 体系
- 新版：`Transaction` + `TransactionDetail` + `InventoryLedger` + `InventorySnapshot` 体系
- 两套体系在页面路由层均被调用，部分接口调用新版服务，部分页面渲染仍指向旧版模板，用户体感不统一

**现存缺陷**：
- `TransactionService.approve_transaction()` 中 `inventory_adjust` 类型的库存调整逻辑使用正 delta（无法表达盘亏），需要在 payload 中用负数 quantity 来表达，API 文档未说明此约定
- 无审批人（approver_id）记录字段，审批完成后无法溯源是谁批准的
- `validate_stock()` 只检查 `SparePart.current_stock`（全局汇总），不区分仓库维度，导致仓库A库存充足但仓库B实际无货时也通过校验

## 二、模块间数据交互评估

### 2.1 数据流向全景图

```
┌─────────────────────────────────────────────────────────────────┐
│                        备件管理模块                              │
│  SparePart (part_code, current_stock, stock_status, ...)       │
└──────────────┬──────────────────────────┬───────────────────────┘
               │ ForeignKey(spare_part_id) │ back_populates
               ▼                           ▼
┌─────────────────────────┐   ┌────────────────────────────────────┐
│     仓库管理模块         │   │         交易管理模块                │
│  Warehouse              │   │  Transaction                       │
│  WarehouseZone          │◄──│  TransactionDetail                 │
│  WarehouseLocation      │   │  InventoryLedger                   │
│  InventoryRecord ←──────┼───┤  InventorySnapshot                 │
│  InboundOrder           │   │                                    │
│  OutboundOrder          │   │  （审批通过 → 写 InventoryLedger   │
└─────────────────────────┘   │   → 调用 _apply_stock）            │
           ▲                   └────────────────────────────────────┘
           │ SQLAlchemy after_insert/update/delete 事件
           │ 自动同步 spare_part.current_stock + warehouse 统计
           │
    InventoryRecord（核心联动表）
    [spare_part_id × warehouse_id × location_id]
```

### 2.2 字段映射一致性问题清单

| 问题类型 | 发生位置 | 错误字段 | 正确字段 | 严重程度 |
|---------|---------|---------|---------|---------|
| 字段名不一致 | `InboundService.create_order()` | `order_number` | `order_no` | 高（运行时报错） |
| 字段名不一致 | `OutboundService.create_order()` | `order_number` | `order_no` | 高（运行时报错） |
| 字段名不一致 | `InboundService.create_order()` | `type` | `inbound_type` | 高（运行时报错） |
| 字段名不一致 | `OutboundService.create_order()` | `type` | `outbound_type` | 高（运行时报错） |
| 字段不存在 | `WarehouseService.get_warehouse_detail_statistics()` | `InventoryRecord.total_value` | `total_amount` | 高（500错误） |
| 字段不存在 | `OutboundService._allocate_*()` | `InventoryRecord.status` | `stock_status` | 高（查询空集） |
| 字段不存在 | `OutboundService._allocate_*()` | `InventoryRecord.entry_date` | `last_inbound_time` 或无此字段 | 高 |
| 旧模型字段 | `WarehouseService.get_total_inventory()` | `Transaction.warehouse_id`, `transaction_type` | `source/target_warehouse_id`, `tx_type` | 高（500错误） |
| 逻辑错误 | `InboundService._log_operation()` | `operator_id.user.real_name`（int无.user属性） | `User.query.get(operator_id).real_name` | 中（日志写入失败） |
| 逻辑错误 | `OutboundService._log_operation()` | 同上 | 同上 | 中 |
| 时区不一致 | `SparePart.created_at/updated_at` | `datetime.now` | `datetime.utcnow` | 低（数据不一致） |
| 并发写入竞争 | `TransactionService._apply_stock()` + `InventoryRecord after_update事件` | 双写 `SparePart.current_stock` | 应只保留一种机制 | 高（数据不一致） |

### 2.3 跨模块数据一致性风险

**风险1：库存双写竞争**

当通过 `TransactionService.approve_transaction()` 审批一笔出库交易时：
1. `_apply_stock()` 直接修改 `SparePart.current_stock`
2. 如果同时有 `InventoryRecord` 发生变化（通过旧版出库流程），`after_update` 事件也会更新 `current_stock`

两条路径可能产生不同的计算结果，最终 `current_stock` 与实际库存不符。

**风险2：库存校验维度缺失**

`TransactionService.validate_stock()` 只校验 `SparePart.current_stock`（全仓库合计），不按仓库校验。实际情况：备件在仓库A有100件、仓库B有0件，从仓库B出库100件会通过校验但实际无法执行。

**风险3：旧版入出库与新版交易的重复计费**

部分页面同时调用旧版 `InboundService/OutboundService`（写 `InboundOrder/OutboundOrder`）和新版 `TransactionService`（写 `Transaction`），相同的一次入库操作可能被记录到两套表中，导致库存统计翻倍。

## 三、行业规范合规性审查

### 3.1 参考规范

- **GB/T 4122**：包装标识国家标准（涉及备件条形码）
- **GB/T 21086-2007**：仓储管理服务规范
- **ISO 55000**：资产管理体系（适用于维修备件管理）
- **MRO管理最佳实践**（GE、西门子、ABB 等工业企业标准）
- **财政部《企业会计准则第1号——存货》**（库存成本核算）
- **供应链运营参考模型（SCOR）**

### 3.2 合规性检查结果

#### 3.2.1 库存成本核算（财务合规）

| 检查项 | 当前状态 | 合规要求 | 差距 |
|-------|---------|---------|------|
| 计价方法 | 未明确（无加权平均/FIFO成本计算逻辑） | 企业会计准则要求选定并一致使用计价方法 | 严重缺失 |
| 出库成本结转 | 仅记录 `unit_cost` 但无自动成本结转 | 出库时应结转库存成本到费用 | 缺失 |
| 库存跌价准备 | 无跌价评估功能 | 期末应评估存货可变现净值 | 缺失 |
| 批次成本追踪 | `InventoryRecord.unit_cost` 存在 | 需与实际采购价格关联 | 部分满足 |

#### 3.2.2 出入库单据完整性

| 检查项 | 当前状态 | 规范要求 | 差距 |
|-------|---------|---------|------|
| 入库单审批 | `InboundOrder` 无审批流程，状态直接从 pending→completed | 应有经办/复核/审批三级 | 缺失 |
| 出库审批 | `OutboundOrder.approval_required` 字段存在但逻辑未完整实现 | 大额/紧急出库应强制审批 | 部分满足 |
| 单据唯一性 | `order_no` 有 UNIQUE 约束 | 满足 | 满足 |
| 操作留痕 | `OperationLog` 表存在，记录 before/after 快照 | 满足 | 满足 |
| 单据撤销限制 | 完成后可撤销（`rollback_inbound/outbound`） | 合规要求完成单据不可随意撤销，应通过红冲 | 不合规 |

#### 3.2.3 仓储管理规范（GB/T 21086）

| 检查项 | 当前状态 | 规范要求 | 差距 |
|-------|---------|---------|------|
| 库区分类 | `WarehouseZone.type`（general/cold/hazardous） | 满足基本要求 | 满足 |
| 危险品管理 | 仅有 type=hazardous 标记，无专项管控 | 危险品需有独立出入库流程、安全规程 | 缺失 |
| 批次管理 | `Batch` 表存在，含生产日期/有效期 | 满足基本批次追踪 | 满足 |
| FEFO合规 | `OutboundService._allocate_fefo()` 已实现 | 满足 | 满足 |
| 盘点频率 | `InventorySnapshot` 表存在，但无计划性盘点调度 | 应支持定期盘点计划 | 部分缺失 |
| 库存冻结 | `locked_quantity` 字段存在 | 满足 | 满足 |

#### 3.2.4 条形码/追踪规范

| 检查项 | 当前状态 | 规范要求 | 差距 |
|-------|---------|---------|------|
| 条形码唯一性 | `SparePart.barcode` UNIQUE可空 | 满足 | 满足 |
| 二维码/RFID | 无 | 行业趋势：RFID库位追踪 | 缺失（建议新增） |
| 序列号管理 | `SerialNumber` 模型存在 | 满足高价值件追踪 | 满足 |

#### 3.2.5 供应链管理（SCOR Model）

| 检查项 | 当前状态 | SCOR要求 | 差距 |
|-------|---------|---------|------|
| 供应商管理 | `Supplier` 表存在，与备件关联 | 满足基础 | 满足 |
| 采购订单关联 | `InboundOrder.purchase_order_id` 仅为 Integer，无外键关联 | 应关联采购订单表 | 部分缺失 |
| 需求预测 | 无 | 基于历史出库数据预测补货需求 | 缺失 |
| 供应商绩效 | 无供应商交货及时率/质量合格率统计 | 应有供应商KPI评估 | 缺失 |

## 四、数据库结构深度分析

### 4.1 核心表关系 ER 图（文字版）

```
Category (1) ──── (N) SparePart (N) ──── (N) InventoryRecord (N) ──── (1) WarehouseLocation
                       |                         |                            |
                Supplier (1) ──── (N)            └── (1) Warehouse            └── (1) WarehouseZone
                       |                                  |                             |
                   Batch (N) ──── (1) SparePart           └── (1) User (manager)        └── (1) Warehouse
                   
Transaction (1) ──── (N) TransactionDetail (1) ──── (1) Batch
     |                         |
     └── (1) User (operator)   └── (N) InventoryLedger
```

### 4.3 事务处理机制分析

#### 现有事务边界

| 操作 | 事务范围 | 问题 |
|-----|---------|------|
| `InboundService.execute_inbound()` | 单次 `db.session.commit()` 涵盖所有明细 | 明细部分失败时整批回滚，`partial` 状态实际不可达 |
| `TransactionService.approve_transaction()` | 单次 `db.session.commit()` | 合理，原子性保证 |
| `InventoryRecord` 事件监听器 | 在原连接上执行原生 SQL | 事件触发时机在 `after_flush` 前后，可能与主事务交叉 |
| `OutboundService.execute_outbound()` | 单次 `db.session.commit()` | 若库存删除后 commit 前崩溃，`location.is_occupied` 未释放 |

#### 关键事务安全问题

**问题1：事件监听器使用原始连接绕过 ORM 事务**

```python
# inventory.py - 事件监听器
def _sync_spare_part_stock(connection, spare_part_id):
    connection.execute(text("UPDATE spare_part SET current_stock = ..."))
```

此处使用的 `connection` 是 ORM 底层连接，绕过了 SQLAlchemy 的 Unit of Work 模式。若 ORM 会话后续回滚，原生 SQL 的更新已提交（SQLite autocommit 模式下），造成数据不一致。

**问题2：缺乏悲观锁（Pessimistic Lock）保护**

出库时扣减库存的操作未使用 `SELECT FOR UPDATE`，高并发场景下两个请求同时读取同一 `InventoryRecord.quantity = 10`，各自扣减 10，最终 quantity = -10。

**问题3：库存记录删除后无法回滚**

`OutboundService.execute_outbound()` 在库存归零时执行 `db.session.delete(inventory)`，若后续出库单撤销（`rollback_outbound`），代码注释已承认"库存记录已被删除，简化处理，continue"，数据永久丢失。

### 4.4 数据库层面的架构性问题

**问题1：三套仓库体系并存**

数据库中 `warehouse`、`warehouse_new`（推测）、`warehouse_v3`（推测）三套表结构可能并存，`dashboard` 路由渲染 `warehouse_new/dashboard.html` 模板说明存在多版本切换。

**问题2：SparePart.current_stock 为派生字段但存为实体字段**

`current_stock` 可以从 `InventoryRecord.quantity` 聚合得出，但作为实体字段存储意味着需要保证时刻同步。当前通过事件监听器 + `_apply_stock` 双通道维护，增加了不一致风险。更好的做法是将其定义为数据库视图或计算属性。

**问题3：Warehouse.total_inventory 等冗余统计字段**

`warehouse` 表中 `total_inventory`、`total_spare_parts`、`utilization_rate` 均为可从 `InventoryRecord` 实时计算的冗余字段，通过事件同步维护。这些字段在高频更新场景下会产生大量 UPDATE 语句，对写入性能有影响。

```python
class ReplenishmentEngine:
    @staticmethod
    def calculate_eoq(annual_demand, order_cost, holding_cost_rate, unit_cost):
        """经济订货量 EOQ = sqrt(2DS/H)"""
        H = unit_cost * holding_cost_rate
        return math.sqrt((2 * annual_demand * order_cost) / H)
    
    @staticmethod
    def calculate_safety_stock(avg_daily_demand, lead_time_days, service_level=0.95):
        """安全库存 = Z * σ * sqrt(L)"""
        z = stats.norm.ppf(service_level)
        # 需要历史出库标准差 σ
        return z * demand_std * math.sqrt(lead_time_days)
```

### D4：供应商绩效管理

**功能描述**：统计供应商交货及时率、质检合格率、价格稳定性，生成供应商评分卡。

```python
class SupplierPerformanceService:
    @staticmethod
    def calculate_delivery_on_time_rate(supplier_id, period_days=90):
        """计算交货及时率"""
        orders = InboundOrder.query.filter(
            InboundOrder.supplier_id == supplier_id,
            InboundOrder.status == 'completed',
            InboundOrder.created_at >= datetime.utcnow() - timedelta(days=period_days)
        ).all()
        
        on_time = sum(1 for o in orders 
                     if o.completed_at and o.expected_date 
                     and o.completed_at.date() <= o.expected_date)
        return (on_time / len(orders) * 100) if orders else 0
    
    @staticmethod
    def calculate_quality_pass_rate(supplier_id, period_days=90):
        """计算质检合格率"""
        orders = InboundOrder.query.filter(
            InboundOrder.supplier_id == supplier_id,
            InboundOrder.quality_check == True,
            InboundOrder.created_at >= datetime.utcnow() - timedelta(days=period_days)
        ).all()
        
        passed = sum(1 for o in orders if o.quality_status == 'passed')
        return (passed / len(orders) * 100) if orders else 0
```

### D5：成本核算模块（加权平均法）

**功能描述**：实现存货的加权平均成本计算，满足财务合规要求。

```python
class CostAccountingService:
    @staticmethod
    def calculate_weighted_avg_cost(spare_part_id, warehouse_id=None):
        """计算加权平均成本"""
        query = db.session.query(
            db.func.sum(InventoryRecord.quantity * InventoryRecord.unit_cost),
            db.func.sum(InventoryRecord.quantity)
        ).filter(InventoryRecord.spare_part_id == spare_part_id)
        
        if warehouse_id:
            query = query.filter(InventoryRecord.warehouse_id == warehouse_id)
        
        total_value, total_qty = query.first()
        if total_qty and total_qty > 0:
            return total_value / total_qty
        return None
    
    @staticmethod
    def record_outbound_cost(transaction_detail_id, method='weighted_avg'):
        """记录出库成本结转"""
        detail = TransactionDetail.query.get(transaction_detail_id)
        if method == 'weighted_avg':
            cost = CostAccountingService.calculate_weighted_avg_cost(
                detail.spare_part_id, 
                Transaction.query.get(detail.transaction_id).source_warehouse_id
            )
            detail.unit_price = cost
            detail.amount = cost * detail.quantity
```

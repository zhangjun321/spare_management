## 用户需求

- 启动当前工作区中的备件管理系统，使系统服务正常运行并可访问。
- 优先使用项目已有的完整启动方式，确保依赖服务、业务服务与页面入口一起可用。
- 若首选启动方式不可用，需要准备可直接切换的备用启动路径。

## 产品概览

- 启动后应能进入现有备件管理系统页面，继续使用仓库管理、交易管理等已存在界面。
- 页面视觉保持当前系统现状，不新增或调整界面设计；重点是让入口可访问、侧边栏与页面联动可正常验证。

## 核心功能

- 启动前检查关键配置与运行环境
- 启动整套服务并确认运行状态
- 验证首页与核心页面入口可访问
- 失败时切换备用启动方案继续提供访问能力

- `d:/Trae/spare_management/app/static/react/index.html` 已存在构建产物入口，可配合集成访问

## Implementation Approach

优先按项目文档与编排文件使用整套容器化启动方式，这与当前项目的依赖关系、访问入口和健康检查设计最一致，能够减少本地环境漂移。若整套方式因 Docker 或端口问题不可用，再回退到本地开发模式，用后端入口和前端开发服务分别启动。

关键决策：

- **首选 Docker 启动**：`DEPLOY.md` 已明确本地开发推荐 `docker compose up -d`，且 `docker-compose.yml` 已把依赖顺序、健康检查和访问端口定义完整。
- **备用本地模式**：`run.py` 与 `frontend/package.json` 已提供独立启动入口，可在容器不可用时恢复访问能力。
- **文档取舍**：`README.md` 存在未解决的 Git 冲突标记，不应作为本次启动的权威依据，应以 `DEPLOY.md`、`docker-compose.yml`、`run.py`、`frontend/package.json` 为准。

性能与可靠性：

- 容器方式下，后端依赖数据库与缓存健康检查后再启动，可降低启动阶段的连接失败概率。
- 启动验证成本为常数级，主要瓶颈在服务初始化时间与端口占用；通过先检查环境、再看日志、最后做页面验证可缩小排查范围。
- 不重复构建、不修改配置，优先复用现有静态产物与既有启动链路，避免额外 I/O 与无关变更。

## Implementation Notes

- 启动前优先确认 `.env` 存在且关键变量完整，避免进入容器后才暴露配置缺失问题。
- 若使用整套容器启动，优先观察 `backend` 与 `nginx` 日志，避免一次性查看全部日志造成噪音。
- 保持现有文件与配置不变，不做无关修复；本次目标仅为安全、可验证地拉起项目。
- 若 Docker 不可用，再切到本地模式，避免双路径同时运行导致端口冲突。

## Architecture Design

- 浏览器访问统一入口后，由前置服务返回静态页面并转发业务请求。
- 业务服务依赖数据库与缓存服务完成初始化后提供页面和接口能力。
- 本地备用模式下，后端单独监听本机端口，前端开发服务通过既有代理对接后端。
